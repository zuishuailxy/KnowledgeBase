## 表操作

### 查询表

```sql
-- create
CREATE TABLE IF NOT EXISTS public.files (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  file_id UUID DEFAULT gen_random_uuid() NOT NULL,
  file_name varchar(255) NOT NULL,
  file_size BIGINT check (file_size > 0) NOT NULL,
  is_public boolean DEFAULT false,
  tags TEXT[],   -- Array of tags for categorization
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 查询
\d  or \d+
\d (表名) //查看对应表的结构

```

### 修改表

```sql
-- add col
ALTER TABLE public.files
ADD COLUMN IF NOT EXISTS description TEXT;  -- Optional description for the file
-- update
ALTER TABLE public.files
ALTER COLUMN updated_at SET DEFAULT CURRENT_TIMESTAMP;
-- rename
ALTER TABLE public.files
RENAME COLUMN is_public TO is_shared;
-- drop
ALTER TABLE public.files DROP COLUMN IF EXISTS tags;
```

### 删除表

```SQL
DROP TABLE IF EXISTS public.files;
-- CASCADE DROP
DROP TABLE IF EXISTS public.files CASCADE;
```

### 清空表

保留表的结构，只是清空数据

```sql
TRUNCATE TABLE public.files;  -- Remove all data from the table without dropping it

-- 清空表并重置自增 id
TRUNCATE TABLE public.files RESTART IDENTITY;  -- Remove all data and reset the identity counter
```

### 特色： 表的继承

创建一个父表，让子表继承它

```sql
CREATE TABLE content (title TEXT, author TEXT);
CREATE TABLE video (duration INT, resolution TEXT) INHERITS (content);  -- Inherit from content table to reuse title and author fields
```

| 查询对象                 | 返回的行            | 返回的列              |
| ------------------------ | ------------------- | --------------------- |
| **子表**（如 `video`）   | 仅子表自己的行      | 父表列 + 子表自己的列 |
| **父表**（如 `content`） | 父表行 + 所有子表行 | 仅父表定义的列        |

因此，**父表查询不会包含子表的额外字段**，这是继承机制的一个重要特点。

## 数据操作

### 插入操作

```sql
INSERT INTO files (file_name, file_size, is_shared, tags, description) VALUES
('example.txt', 1024, false, ARRAY['text', 'document'], 'An example text file.'),
('photo.jpg', 2048000, true, ARRAY['image', 'photo'], 'A high-resolution photo.'),
('presentation.pptx', 5120000, false, ARRAY['presentation', 'slides'], 'A business presentation.');
```

### 查询

```sql
select * from files where file_size > 1000000 AND is_shared = true;


SELECT * FROM files WHERE file_name LIKE '%.jpg';

-- 数组查询 包含 下面两者都行
SELECT * FROM files WHERE 'image' = ANY(tags);  -- Query files that have 'image' tag
SELECT * FROM files WHERE tags @> ARRAY['image'];  -- Query files that have 'image' tag using array containment operator

-- 查询 tags 包含 'text' 或 'slides' 任一标签的记录
SELECT * FROM files WHERE tags && ARRAY['text', 'slides'];

-- 数组完全等于某个值
SELECT * FROM files WHERE tags = ARRAY['image', 'photo'];

-- 查询 tags 数组长度大于 2 的记录
SELECT * FROM files WHERE array_length(tags, 1) > 2;
-- 注意：如果 tags 为 NULL，array_length 返回 NULL，需要用 COALESCE
SELECT * FROM files WHERE COALESCE(array_length(tags, 1), 0) > 2;

-- 聚合查询，groupby
SELECT is_shared, count(*), SUM(array_length(tags, 0)) FROM files GROUP BY is_shared;

-- 数据转换与格式化
select LEFT(file_id::TEXT, 8) as short_id, file_name, to_char(created_at, 'YYYY-MM-DD') as simple_date from files;
```

### Update

```sql
Update files set updated_at = CURRENT_TIMESTAMP where id = 1;  -- Update the timestamp for a specific file

-- 更新多个值
UPDATE files SET is_shared = true, file_size = file_size + 1024 WHERE file_name = 'example.txt';  -- Update multiple columns for a specific file
```

### Delete

```sql
DELETE FROM files WHERE file_size < 1024;
```

### 创建 + 回显

```sql
INsert into files (file_name, file_size) VALUES
('notes.txt', 512) RETURNING file_id, created_at;  -- Insert a new file and return its file_id and created_at
```

## JSONB

```sql
-- 建表
CREATE TABLE products (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name TEXT NOT NULL,
    attributes JSONB   -- 存储产品属性，如颜色、尺寸、重量等
);

-- insert
INSERT INTO products (name, attributes)
VALUES ('T恤', '{"color": "红色", "size": "M", "weight_kg": 0.15}');
```

### 基础查询

读取 JSONB 中的某个字段（-> 返回 JSON，->> 返回文本）

```sql
-- 获取颜色（文本）
SELECT name, attributes->>'color' AS color FROM products;

-- 获取整个 attributes 中的 size 字段（JSON 格式）
SELECT name, attributes->'size' FROM products;
```

### 条件过滤

```sql
-- 找出颜色为红色的产品
SELECT * FROM products WHERE attributes->>'color' = '红色';

-- 找出重量大于 0.1kg 的产品（注意：->> 返回文本，需强转数值）
SELECT * FROM products
WHERE (attributes->>'weight_kg')::DECIMAL > 0.1;

-- 包含 color 字段的
SELECT * FROM products WHERE attributes ? 'color';
```

### 嵌套对象与数组

```sql
-- 插入包含数组和嵌套对象的数据
INSERT INTO products (name, attributes) VALUES (
    '手机',
    '{
        "brand": "X",
        "specs": {"ram": "8GB", "storage": "128GB"},
        "colors": ["黑色", "蓝色"]
    }'
);

-- 查询嵌套字段
SELECT attributes->'specs'->>'ram' AS ram FROM products WHERE name = '手机';

-- 查询数组是否包含某个元素（JSONB 包含运算符 @>）
SELECT * FROM products
WHERE attributes->'colors' @> '"黑色"'::jsonb;
```

### 更新 JSONB 字段

```sql
-- 5.1 整体替换
UPDATE products SET attributes = '{"color": "绿色", "size": "L"}' WHERE name = 'T恤' AND id = 1;

-- 5.2 修改/新增某个键（使用 `||` 连接符）
UPDATE products
SET attributes = attributes || '{"discount": 0.1}'::jsonb
WHERE name = '手机';

UPDATE products
SET attributes = attributes - 'weight_kg'
WHERE name = 'T恤';

-- 5.3 删除某个键（`-` 操作符）
UPDATE products
SET attributes = attributes - 'weight_kg'
WHERE name = 'T恤';

-- 5.4 更深层的修改（使用 jsonb_set）
-- 把手机的 storage 改为 "256GB"
UPDATE products
SET attributes = jsonb_set(
    attributes,                     -- 原 JSONB
    '{specs,storage}',              -- 路径
    '"512GB"'::jsonb,              -- 新值
    true                            -- 若不存在是否创建
)
WHERE name = '手机';
```

## 索引

索引 是一种用于加速数据检索的辅助结构。

### 为什么需要索引

- 提升查询速度：对 WHERE、JOIN、ORDER BY 涉及的列建索引，可将查询耗时从 O(n) 降为 O(log n) 甚至 O(1)。
- 保证唯一性：UNIQUE 索引自动保证列值不重复。
- 加速排序和分组。

代价：

- 占用额外存储空间（索引本身也是数据）
- 降低写入性能（INSERT/UPDATE/DELETE 时也要同步更新索引）。

因此，索引适合读多写少且查询频繁的场景。

### 索引的工作原理

最常见的 B-Tree， 是一种平衡多路搜索树

### 如何使用

#### 创建索引

① B-Tree 索引（默认，适合等值、范围、排序）

```sql
-- 单列索引
CREATE INDEX idx_users_email ON users (email);

-- 多列复合索引
CREATE INDEX idx_users_name_age ON users (last_name, age);
```

② 唯一索引（自动创建 UNIQUE 约束）

```sql
CREATE UNIQUE INDEX idx_users_email_uniq ON users (email);
```

③ 表达式索引（在函数或表达式上建索引）

```sql
-- 加速 WHERE lower(email) = '...' 查询
CREATE INDEX idx_users_lower_email ON users (lower(email));
```

④ 部分索引（只对满足条件的行建索引）

```sql
-- 只为 active 用户建索引
CREATE INDEX idx_active_users_email ON users (email) WHERE is_active = true;
```

⑤ JSONB 的 GIN 索引（支持 @>、? 等操作）

```sql
CREATE INDEX idx_products_attrs ON products USING GIN (attributes);
```

⑥ 数组的 GIN 索引

```sql
CREATE INDEX idx_files_tags ON files USING GIN (tags);
```

#### 查看索引

```sql
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'products';
```

#### 删除索引

```sql
DROP INDEX idx_users_email;
```

#### 验证索引是否被使用（执行计划）

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM users WHERE email = 'alice@example.com';
```

- 出现 Index Scan 或 Bitmap Index Scan 说明使用了索引。
- 出现 Seq Scan 说明没有使用索引（可能表太小、条件过滤性差等）。

### 最佳实践

1. 为高频查询的 WHERE 列建索引，尤其是选择性高的列（值分布广，如 ID、邮箱）。
2. 谨慎使用复合索引：把等值条件列放前面，范围条件列放后面。
3. 避免索引冗余：已有 (a,b) 索引，不必再建 (a) 索引（前者可以覆盖后者）。
4. 不要在小表上建索引（全表扫描更快）。
5. 定期维护：通过 REINDEX 重建膨胀的索引；用 VACUUM 回收空间。
6. 监控未使用的索引：通过 pg_stat_user_indexes 看使用次数，删除长期不用的索引。

| `LAG(expr, offset)`       | 取当前行前offset行的值    | `LAG(salary, 1) OVER (ORDER BY hire_date)`                   |

---

## 🧪 示例：组合使用

```sql
-- 将 tags 数组转为逗号分隔的字符串
SELECT id, array_to_string(tags, ',') FROM files;

-- 统计每个标签出现的次数（将数组展开）
SELECT tag, COUNT(*) FROM files, unnest(tags) AS tag GROUP BY tag;
```

### 窗口函数

> 窗口函数可以在“不合并行”的情况下，对每一行数据计算一个“基于其他行”的值

而普通的 GROUP BY 聚合会把多行压缩成一行，丢失细节。

```sql
SELECT salesperson, product, amount,
       SUM(amount) OVER (PARTITION BY salesperson) AS person_total
FROM sales;
```

#### 窗口函数的基本用法

```sql
函数() OVER (
    PARTITION BY 分组列   -- 可选：定义“窗口”范围（类似 GROUP BY）
    ORDER BY 排序列      -- 可选：定义窗口内顺序
    ROWS/RANGE 子句      -- 可选：限定窗口边界（更高级）
)
```

函数() 可以是：

- 聚合函数：SUM、AVG、COUNT、MAX、MIN

- 专用窗口函数：ROW_NUMBER()、RANK()、DENSE_RANK()、LAG()、LEAD() 等

> 窗口函数 = 聚合 + 不合并行 + 可以排序/排名/引用前后行。

它让你在保留原始行数据的同时，还能进行复杂的跨行计算。

## 多表关系

多表关系指的是通过表之间的逻辑关联（如外键约束）和查询操作（如 JOIN）来组合数据的方式。

在 PostgreSQL 中，**多表关系**指的是通过表之间的逻辑关联（如外键约束）和查询操作（如 JOIN）来组合数据的方式。正确地设计和管理表关系是关系型数据库的核心。

### 一、关系类型（逻辑层面）

| 关系类型         | 说明                               | 例子                      |
| ---------------- | ---------------------------------- | ------------------------- |
| **一对一 (1:1)** | 表 A 的一条记录对应表 B 的一条记录 | 用户表 → 用户详情表       |
| **一对多 (1:N)** | 表 A 的一条记录对应表 B 的多条记录 | 一个作者 → 多本书         |
| **多对多 (M:N)** | 表 A 的多条记录对应表 B 的多条记录 | 学生 ↔ 课程（通过中间表） |

---

### 二、物理实现：外键约束（`FOREIGN KEY`）

外键是 PostgreSQL 中最常用的保证**参照完整性**的机制。

### 1. 一对多示例：作者 → 书籍

```sql
CREATE TABLE authors (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name TEXT NOT NULL
);

CREATE TABLE books (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    title TEXT NOT NULL,
    author_id BIGINT REFERENCES authors(id) ON DELETE CASCADE
);
```

- `REFERENCES authors(id)` 声明外键，确保 `author_id` 必须存在于 `authors.id` 中。
- `ON DELETE CASCADE`：作者被删除时，其所有书籍自动删除。其他选项：`SET NULL`、`SET DEFAULT`、`RESTRICT`。

### 2. 多对多示例：学生 ↔ 课程（中间表）

```sql
CREATE TABLE students ( id BIGINT PRIMARY KEY, name TEXT );
CREATE TABLE courses ( id BIGINT PRIMARY KEY, title TEXT );

-- 中间表，存储学生与课程的选课关系
CREATE TABLE enrollments (
    student_id BIGINT REFERENCES students(id),
    course_id BIGINT REFERENCES courses(id),
    enrolled_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (student_id, course_id)   -- 复合主键，防止重复选课
);
```

### 3. 一对一示例：用户与扩展资料

```sql
CREATE TABLE users ( id BIGINT PRIMARY KEY, username TEXT );
CREATE TABLE user_profiles (
    user_id BIGINT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    avatar_url TEXT,
    bio TEXT
);
```

通常将不常用的字段或大字段拆到另一张表，且 `user_id` 既是主键又是外键。

---

## 三、查询多表数据：`JOIN` 系列

有了外键关系，就可以通过 `JOIN` 将多张表的数据横向合并。

### 常用 JOIN 类型

| JOIN 类型    | 说明                                |
| ------------ | ----------------------------------- |
| `INNER JOIN` | 只返回两表匹配的行                  |
| `LEFT JOIN`  | 返回左表所有行，右表不匹配处为 NULL |
| `RIGHT JOIN` | 返回右表所有行，左表不匹配处为 NULL |
| `FULL JOIN`  | 返回两表所有行，不匹配处为 NULL     |
| `CROSS JOIN` | 笛卡尔积（极少用）                  |

### 示例：查询每本书及其作者

```sql
SELECT books.title, authors.name
FROM books
JOIN authors ON books.author_id = authors.id;
```

### 示例：多对多查询（选修了“数学”课程的所有学生）

```sql
SELECT students.name
FROM students
JOIN enrollments ON students.id = enrollments.student_id
JOIN courses ON enrollments.course_id = courses.id
WHERE courses.title = '数学';
```

## 四、管理外键行为

在定义外键时，可以指定 `ON DELETE` 和 `ON UPDATE` 的动作：

| 动作                     | 含义                                     |
| ------------------------ | ---------------------------------------- |
| `CASCADE`                | 父表删除/更新时，子表相应行自动删除/更新 |
| `SET NULL`               | 父表删除/更新时，子表外键列设为 NULL     |
| `SET DEFAULT`            | 设为默认值（需有默认值）                 |
| `RESTRICT` / `NO ACTION` | 阻止删除/更新（默认行为，防止孤儿数据）  |

```sql
ALTER TABLE books
ADD CONSTRAINT fk_author
FOREIGN KEY (author_id) REFERENCES authors(id)
ON DELETE SET NULL ON UPDATE CASCADE;
```

## 五、高级关系特性

### 1. 表继承（`INHERITS`）

PostgreSQL 独有的特性，子表自动继承父表的所有列（之前讨论过）。

```sql
CREATE TABLE content (title TEXT, author TEXT);
CREATE TABLE video (duration INT) INHERITS (content);
```

查询父表会返回子表行，但只带父表的列。很少用于传统外键关系，更多用于分区。

### 2. 分区表（`PARTITION BY`）

将一个大表按某种规则拆分为多个物理子表，对外仍视为一张表。常用于时序数据。

```sql
CREATE TABLE measurement (city TEXT, logdate DATE, temp INT)
PARTITION BY RANGE (logdate);
```

分区与“关系”概念不同，但同样涉及父子表结构。

### 3. 外部数据包装器（`postgres_fdw`）

可以访问远程 PostgreSQL 或其他数据库的表，实现跨数据库的“关系”。

-- 查看某个表的外键约束
\d books

-- 查询所有外键约束（系统表）

    tc.constraint_name,
    tc.table_name,
    kcu.column_name,

    ccu.column_name AS foreign_column_name

## 七、最佳实践与注意事项

- **务必显式定义外键**：保证数据一致性，也让数据库优化器能生成更好的执行计划。
- **索引外键列**：虽然 PostgreSQL 不会自动为外键列创建索引，但建议手动创建，加速 `JOIN` 和级联操作。
  ```sql
  CREATE INDEX idx_books_author_id ON books(author_id);
  ```
- **避免级联删除过于激进**：`ON DELETE CASCADE` 可能误删大量数据，生产环境谨慎使用。
- **多对多必须用中间表**：不要试图用数组或 JSON 存储多对多关系，会破坏引用完整性和查询性能。
- **考虑使用逻辑复制或物化视图**：若需跨库关联查询，优先考虑 `postgres_fdw` 或 ETL。

## 八、总结

PostgreSQL 中的多表关系通过 **外键约束** 定义完整性，通过 **`JOIN` 查询** 获取组合数据。理解 1:1、1:N、M:N 的转化方法（尤其是中间表），以及 `ON DELETE` / `ON UPDATE` 行为，就能构建健壮的关系数据模型。对于特殊的继承或分区需求，PostgreSQL 也提供了扩展特性，但日常开发最常用的仍是标准的外键 + 连接查询。

## join

`JOIN` 是关系型数据库最核心的操作之一，它允许你根据关联条件将两个或多个表中的数据横向合并成一张虚拟的结果表。

### 为什么需要 JOIN？

在规范化的数据库中，数据被分散存储在不同的表中以减少冗余。例如：

- users 表存用户基本信息
- orders 表存订单，通过 user_id 关联用户

当你需要同时查询“用户姓名”和“他的所有订单金额”时，就必须用 JOIN 把两张表连接起来。

### JOIN 的核心术语

- **左表、右表**：JOIN 关键字左边的表和右边的表。
- **连接条件**：用于匹配行的表达式，通常形如 left_table.key = right_table.foreign_key。
- **结果集**：两表按条件组合后的临时表。

### JOIN 的种类及行为

假设有两张简单的表：

```sql
CREATE TABLE students (
    id INT PRIMARY KEY,
    name TEXT
);

CREATE TABLE scores (
    student_id INT,
    score INT
);

INSERT INTO students VALUES (1, 'Alice'), (2, 'Bob'), (3, 'Charlie');
INSERT INTO scores VALUES (1, 90), (2, 85), (4, 88);   -- 注意 student_id=4 在 students 中不存在
```

1. INNER JOIN —— 最常见

只返回两表中连接条件匹配的行，不匹配的行全部丢弃。

```sql
SELECT students.name, scores.score
FROM students
INNER JOIN scores ON students.id = scores.student_id;
```

2. LEFT JOIN (LEFT OUTER JOIN) —— 最实用

返回左表所有行，右表没有匹配的列补 NULL。

```sql
SELECT students.name, scores.score
FROM students
LEFT JOIN scores ON students.id = scores.student_id;
```

3. RIGHT JOIN

与 LEFT JOIN 对称，返回右表所有行。通常可以通过交换表顺序用 LEFT JOIN 代替，所以实际使用较少。

```sql
SELECT students.name, scores.score
FROM students
RIGHT JOIN scores ON students.id = scores.student_id;
```

4. FULL JOIN (FULL OUTER JOIN)

返回两表所有行，任一表无匹配则另一侧补 NULL。

```sql
SELECT students.name, scores.score
FROM students
FULL JOIN scores ON students.id = scores.student_id;
```

5. CROSS JOIN —— 笛卡尔积

不指定连接条件，左表每一行与右表每一行组合。结果行数 = 左表行数 × 右表行数，几乎只用在对维度组合的特定场景。

```sql
SELECT students.name, scores.score
FROM students
CROSS JOIN scores;
```

### 自连接

一张表与自身连接，通常用于处理层级数据或比较同一表内不同行。

示例：查询每个员工的上级姓名（员工表包含 manager_id）

```sql
CREATE TABLE employees (id INT, name TEXT, manager_id INT);
INSERT INTO employees VALUES (1, 'Alice', NULL), (2, 'Bob', 1), (3, 'Charlie', 1);

SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
```

### 多表连接

示例：学生、选课（enrollments）、课程（courses）三表联合。

```sql
SELECT s.name, c.title
FROM students s
JOIN enrollments e ON s.id = e.student_id
JOIN courses c ON e.course_id = c.id;
```

PostgreSQL 会按顺序（或根据优化器判断）执行两两连接。注意前面的 JOIN 结果当作虚拟表再与下一张表连接。

### PostgreSQL 独有的 JOIN 类型

**LATERAL JOIN**

子查询可以引用前面的表中的列。常用于对每行执行一个子查询并展开结果。

```sql
-- 对每个学生，查询其最新的 2 条成绩记录
SELECT s.name, latest.score
FROM students s,
LATERAL (
    SELECT score FROM scores
    WHERE student_id = s.id
    ORDER BY exam_date DESC
    LIMIT 2
) latest;
```

### JOIN 的性能优化与注意事项

**1. 索引是关键**

- 连接条件列（尤其是外键）应该建立索引：
  `CREATE INDEX idx_scores_student_id ON scores(student_id);`

- 否则数据库会对大表执行顺序扫描 + 嵌套循环，代价极高。

**2. 驱动表的选择（连接顺序）**

- PostgreSQL 的查询优化器会自动选择小表作为驱动表，但仍建议先用 WHERE 条件过滤数据，减少连接规模。
- 可以通过 `EXPLAIN (ANALYZE, BUFFERS) `查看实际连接方式（Nested Loop, Hash Join, Merge Join）。

**3. 避免在连接条件上使用函数**

```sql
-- 坏：无法使用 scores.student_id 上的索引
SELECT * FROM students
JOIN scores ON lower(students.name) = lower(scores.student_name);
```

应改为表达式索引或调整应用逻辑。

**4. 只 SELECT 需要的列**

`SELECT *` 会额外传输无用数据，增加 I/O 和内存开销。

**5. 考虑是否存在更优的连接顺序**

通常优化器会处理，但遇到复杂多表连接且统计信息不准时，可以手动调整 SQL 顺序（用括号）

### 实战示例

```sql
-- 查询每个作者及其书的数量（包括没有书的作者）
SELECT a.name, COUNT(b.id) AS book_count
FROM authors a
LEFT JOIN books b ON a.id = b.author_id
GROUP BY a.id, a.name;
```

### 常见错误与陷阱

1. **忘记连接条件** → 变成了 CROSS JOIN，结果暴增。

```sql
-- 错误
SELECT * FROM students, scores;
```

2. 连接条件使用 = 之外的操作符（如 <, >），可能导致 Index Scan 变为 Seq Scan，需谨慎
3. 重复行：如果右表有多条匹配，左表行会重复出现。这是正常现象，不确定时可以加 DISTINCT，但最好先理解业务
4. NULL 值的陷阱：外键列允许 NULL 时，INNER JOIN 不会匹配 NULL，LEFT JOIN 则会保留左表且右表部分为 NULL。

## 📖 数据库事务详解（以 PostgreSQL 为例）

**事务**是数据库执行过程中的一个**逻辑工作单元**，它包含一组操作（比如多条 `INSERT`、`UPDATE`、`DELETE`），这些操作要么**全部成功提交**，要么**全部失败回滚**，不会出现部分完成的情况。

可以把事务想象成一个**原子操作** —— 就像银行转账：扣钱和加钱必须同时成功，否则整个操作撤销，保持数据一致。

---

## 一、事务的四大特性（ACID）

| 特性                     | 含义                                                                         | 比喻                                             |
| ------------------------ | ---------------------------------------------------------------------------- | ------------------------------------------------ |
| **原子性 (Atomicity)**   | 事务中的所有操作是一个不可分割的整体，要么全做，要么全不做                   | 转账：扣钱和加钱必须同时完成，不能只扣不加       |
| **一致性 (Consistency)** | 事务执行前后，数据库从一个一致状态变为另一个一致状态（不破坏约束、业务规则） | 转账后总金额不变；外键约束、唯一约束等不会被破坏 |
| **隔离性 (Isolation)**   | 并发执行的事务之间互不干扰，每个事务都感觉自己在独立运行                     | 两个人同时改同一个账户，结果应该等价于串行执行   |
| **持久性 (Durability)**  | 一旦事务提交，其对数据库的修改是永久的，即使系统崩溃也不会丢失               | 转账成功后，钱不会无缘无故消失                   |

---

## 二、事务控制语句（PostgreSQL）

```sql
-- 开始一个事务
BEGIN;   -- 或 START TRANSACTION;

-- 执行一些 DML 操作
INSERT INTO accounts (id, balance) VALUES (1, 1000);
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- 如果一切正常，提交事务（使修改永久生效）
COMMIT;

-- 如果出现问题，撤销所有修改
ROLLBACK;
```

### 保存点（SAVEPOINT）—— 部分回滚

```sql
BEGIN;
  INSERT INTO logs (msg) VALUES ('Step 1');
  SAVEPOINT step1;
    INSERT INTO logs (msg) VALUES ('Step 2');
    -- 发现 Step 2 有问题，回滚到保存点
    ROLLBACK TO SAVEPOINT step1;
  INSERT INTO logs (msg) VALUES ('Step 3');
COMMIT;   -- 最终只会插入 'Step 1' 和 'Step 3'
```

## 三、并发事务引发的问题

当多个事务同时执行且隔离性不足时，可能出现以下异常：

| 问题                                 | 说明                                                                 | 例子                                                              |
| ------------------------------------ | -------------------------------------------------------------------- | ----------------------------------------------------------------- |
| **脏读 (Dirty Read)**                | 事务A读取了事务B尚未提交的修改，若B回滚，A读到了“脏”数据             | A看到B转账未提交的-100，以为账户扣了，但B最终回滚                 |
| **不可重复读 (Non-repeatable Read)** | 同一事务内两次读取同一行数据，得到不同结果（因为其他事务提交了修改） | 事务内第一次查余额是100，第二次查变成90（另一个事务扣了款并提交） |
| **幻读 (Phantom Read)**              | 同一事务内两次查询满足条件的行数不同（其他事务插入了新行）           | 第一次查账户数为2，第二次查变成3（另一个事务新增了账户）          |

---

## 四、隔离级别（ISO SQL 标准）

PostgreSQL 支持四种隔离级别，但**默认且最常用的是 `READ COMMITTED`**。它实际上只实现了三种（`READ UNCOMMITTED` 会被当作 `READ COMMITTED` 处理）。

| 隔离级别                  | 脏读   | 不可重复读 | 幻读           | 并发性能                        |
| ------------------------- | ------ | ---------- | -------------- | ------------------------------- |
| `READ UNCOMMITTED`        | 可能   | 可能       | 可能           | 最高（但 PG 不支持，实际为 RC） |
| `READ COMMITTED` (PG默认) | 不可能 | 可能       | 可能           | 高                              |
| `REPEATABLE READ`         | 不可能 | 不可能     | 不可能（PG中） | 中等                            |
| `SERIALIZABLE`            | 不可能 | 不可能     | 不可能         | 最低（严格串行化）              |

### 设置隔离级别

```sql
-- 会话级
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务级
BEGIN ISOLATION LEVEL SERIALIZABLE;
-- ... 操作 ...
COMMIT;
```

**注意**：PostgreSQL 的 `REPEATABLE READ` 级别实际上**避免了幻读**（通过快照隔离），比标准 SQL 更严格。`SERIALIZABLE` 使用谓词锁，可能因序列化失败而报错，需要重试。

## 五、PostgreSQL 事务的特点

### 1. 自动提交模式

默认情况下，每条单独的 SQL 语句会被包装成一个隐式事务（执行完自动提交）。可以通过 `SET AUTOCOMMIT OFF;` 或显式使用 `BEGIN` 来关闭。

### 2. 事务中 DDL（表结构修改）也是可回滚的

大多数数据库（如 MySQL）不支持 DDL 回滚，但 **PostgreSQL 支持**：

```sql
BEGIN;
  CREATE TABLE temp (id int);
  ALTER TABLE temp ADD COLUMN name text;
ROLLBACK;   -- 表被完全撤销，好像从未创建过
```

### 3. 错误处理与事务状态

如果事务中某条语句出错，PostgreSQL 不会自动回滚整个事务，但该事务会进入**中断状态**，后续操作会被拒绝，直到执行 `ROLLBACK` 或 `COMMIT`（通常建议回滚后重试）。

```sql
BEGIN;
  INSERT INTO accounts (id) VALUES (1);   -- 成功
  INSERT INTO accounts (id) VALUES (1);   -- 违反唯一约束，出错
  INSERT INTO accounts (id) VALUES (2);   -- 这条不会再执行（事务已中断）
ROLLBACK;   -- 必须回滚才能恢复正常
```

### 4. 保存点错误恢复

可以使用保存点捕获错误并继续事务：

```sql
BEGIN;
  INSERT INTO accounts VALUES (1);
  SAVEPOINT my_save;
    INSERT INTO accounts VALUES (1);   -- 出错
    ROLLBACK TO SAVEPOINT my_save;     -- 回滚到保存点，保留前面的插入
  INSERT INTO accounts VALUES (2);
COMMIT;   -- 最终插入了 1 和 2
```

## 六、实际应用场景与示例

### 场景1：银行转账（经典事务）

```sql
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
  -- 可在此检查余额是否 >=0，否则 ROLLBACK
  -- 假设一切正常
COMMIT;
```

### 场景2：订单与库存（多表一致性）

```sql
BEGIN;
  -- 1. 插入订单
  INSERT INTO orders (user_id, product_id, quantity) VALUES (1, 100, 2);
  -- 2. 扣减库存
  UPDATE products SET stock = stock - 2 WHERE id = 100 AND stock >= 2;
  -- 检查受影响行数，如果为0说明库存不足，ROLLBACK
  GET DIAGNOSTICS rows = ROW_COUNT;
  IF rows = 0 THEN
    ROLLBACK;
    RAISE EXCEPTION '库存不足';
  END IF;
COMMIT;
```

### 场景3：重试序列化失败

```sql
-- 在高隔离级别下可能因并发冲突失败，应用层应重试
DO $$
DECLARE
  retries INT := 3;
BEGIN
  LOOP
    BEGIN
      SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
      -- 执行复杂业务...
      COMMIT;
      EXIT;  -- 成功则退出循环
    EXCEPTION
      WHEN SQLSTATE '40001' THEN  -- 序列化失败
        retries := retries - 1;
        IF retries = 0 THEN RAISE; END IF;
        PERFORM pg_sleep(0.1 * random());
    END;
  END LOOP;
END $$;
```

## 七、长事务与性能注意事项

- **长事务有害**：持有锁、阻止 VACUUM 清理死元组、导致数据库膨胀。
- **避免在事务中做交互等待**（如等待用户输入）。
- **尽量让事务短小精悍**，只包含必要的操作。
- **使用 `pg_stat_activity` 监控长事务**：
  ```sql
  SELECT pid, age(clock_timestamp(), xact_start) AS duration, query
  FROM pg_stat_activity
  WHERE state = 'active' AND xact_start IS NOT NULL
  ORDER BY duration DESC;
  ```

| 默认单个 SQL 自动提交       | 无需任何操作                        |
| 将多条 SQL 作为一个原子单元 | `BEGIN; ... COMMIT;`                |
| 出错时撤销所有操作          | `ROLLBACK`                          |
| 部分回滚                    | `SAVEPOINT ... ROLLBACK TO ...`     |

| 避免不可重复读和幻读        | 使用 `REPEATABLE READ`              |

| 查看当前隔离级别            | `SHOW TRANSACTION ISOLATION LEVEL;` |

事务是数据库保证数据一致性的基石。在 PostgreSQL 中，事务不仅支持标准 ACID，还允许 DDL 回滚和保存点，非常强大。开发时请始终考虑在合适的地方使用显式事务，确保业务操作的原子性。

## 🔒 数据库锁详解（以 PostgreSQL 为例）

**锁**是数据库并发控制的核心机制，用于协调多个事务对同一数据的访问，防止出现数据不一致、脏读、幻读等问题。简单说：**锁就是给数据加上一个“正在使用”的标记，其他事务必须等待或绕过**。

锁分为两类目标：**控制对表结构的修改（DDL）** 和 **控制对行数据的读写（DML）**。PostgreSQL 主要提供表级锁和行级锁，并通过 **MVCC（多版本并发控制）** 减少锁的争用。

---

## 一、锁的基本类型

### 1. 表级锁（Table-level Locks）

锁定整个表，粒度粗，并发性低，通常由 DDL 或某些显式命令产生。自动使用。

| 锁模式                   | 冲突模式                                                                                                                | 典型触发语句                                                      | 说明                                       |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- | ------------------------------------------ |
| `ACCESS SHARE`           | `ACCESS EXCLUSIVE`                                                                                                      | `SELECT`                                                          | 读表，阻止并发的 `TRUNCATE` / `REINDEX` 等 |
| `ROW SHARE`              | `EXCLUSIVE`, `ACCESS EXCLUSIVE`                                                                                         | `SELECT FOR UPDATE/SHARE`                                         | 表明打算更新某些行                         |
| `ROW EXCLUSIVE`          | `SHARE`, `SHARE ROW EXCLUSIVE`, `EXCLUSIVE`, `ACCESS EXCLUSIVE`                                                         | `INSERT`, `UPDATE`, `DELETE`                                      | 修改表数据                                 |
| `SHARE UPDATE EXCLUSIVE` | `SHARE UPDATE EXCLUSIVE`, `SHARE`, `SHARE ROW EXCLUSIVE`, `EXCLUSIVE`, `ACCESS EXCLUSIVE`                               | `VACUUM`, `CREATE INDEX CONCURRENTLY`                             | 保护表结构不被并发破坏                     |
| `SHARE`                  | `ROW EXCLUSIVE`, `SHARE UPDATE EXCLUSIVE`, `SHARE ROW EXCLUSIVE`, `EXCLUSIVE`, `ACCESS EXCLUSIVE`                       | `CREATE INDEX`（非并发）                                          | 允许并发读，禁止修改                       |
| `SHARE ROW EXCLUSIVE`    | 所有其他锁除了 `ACCESS SHARE`                                                                                           | 少见，由某些插件触发                                              | 比 `SHARE` 更严格，禁止读以外的操作        |
| `EXCLUSIVE`              | `ROW SHARE`, `ROW EXCLUSIVE`, `SHARE UPDATE EXCLUSIVE`, `SHARE`, `SHARE ROW EXCLUSIVE`, `EXCLUSIVE`, `ACCESS EXCLUSIVE` | `REFRESH MATERIALIZED VIEW CONCURRENTLY`                          | 允许并发 `ACCESS SHARE`，禁止其他一切      |
| `ACCESS EXCLUSIVE`       | **所有锁**                                                                                                              | `DROP TABLE`, `TRUNCATE`, `REINDEX`, `VACUUM FULL`, `ALTER TABLE` | 完全独占表，甚至阻止 `SELECT`              |

**注意**：`SELECT` 只获取 `ACCESS SHARE`，所以两个 `SELECT` 不会互相阻塞。但 `SELECT` 会阻塞 `TRUNCATE`，因为 `TRUNCATE` 需要 `ACCESS EXCLUSIVE`。

---

### 2. 行级锁（Row-level Locks）

只锁定被操作的行，并发性好。行级锁不直接存在，而是通过 **行上的元组（tuple）的版本可见性** 和 **事务 ID 的标记** 来实现。

| 行锁模式            | 说明                                                         | 产生语句                |
| ------------------- | ------------------------------------------------------------ | ----------------------- |
| `FOR UPDATE`        | 排他锁，阻止其他事务对这些行加任何行锁（包括读锁）           | `SELECT ... FOR UPDATE` |
| `FOR NO KEY UPDATE` | 类似 `FOR UPDATE`，但只阻止加 `FOR UPDATE` / `FOR KEY SHARE` | `UPDATE`（不更新键列）  |
| `FOR SHARE`         | 共享锁，允许其他事务加 `FOR SHARE` 但阻止 `FOR UPDATE`       | `SELECT ... FOR SHARE`  |
| `FOR KEY SHARE`     | 最弱的锁，只阻止其他事务更改键列（用于外键检查）             | 外键约束的检查过程      |

**行为总结**：

- `FOR UPDATE` —— 最严格，可防止行被任何修改。
- `FOR NO KEY UPDATE` —— 允许并发 `FOR SHARE` 但阻止 `FOR UPDATE`。
- `FOR SHARE` —— 允许其他 `FOR SHARE`，但不能修改行（包括更新、删除）。
- `FOR KEY SHARE` —— 允许除键列更新之外的所有操作。

---

## 二、MVCC（多版本并发控制）—— PostgreSQL 不需要“读锁”的秘密

PostgreSQL 采用 **MVCC** 实现高并发读写，**读操作不加锁**（仅获取 `ACCESS SHARE` 表级锁，不阻碍写）。核心原理：

- 每个更新操作产生新版本（元组），旧版本仍保留。
- 每个事务看到的是**事务开始瞬间的快照**（基于事务 ID 和可见性规则）。
- `DELETE` / `UPDATE` 不会真正删除旧数据，只是标记为“无效”。

因此：

- **读不阻塞写，写不阻塞读**（真正的并发）。
- 仅当两个事务尝试**同时修改同一行**时，才需要行级锁（先修改的事务获得锁，后修改的事务等待或超时）。

---

## 三、显式锁与隐式锁

### 隐式锁

- 普通 `INSERT`, `UPDATE`, `DELETE` 会自动在受影响的表上获取 `ROW EXCLUSIVE` 表锁，在受影响的行上获取适当的行锁（通常是 `FOR NO KEY UPDATE` 或更强的锁）。
- `SELECT` 自动获取 `ACCESS SHARE`。

### 显式锁

用户手动控制：

```sql
-- 表级锁
LOCK TABLE mytable IN ACCESS EXCLUSIVE MODE;

-- 行级锁（在 SELECT 上加）
SELECT * FROM orders WHERE id = 10 FOR UPDATE;
```

**使用场景**：需要防止其他事务并发修改某些行时，或在事务中多次读取同一行并要求数据不变化时（尽管 MVCC 已保证可重复读，但 `FOR UPDATE` 可阻止其他事务提交干扰）。

---

## 四、死锁（Deadlock）

**死锁**：两个或多个事务相互等待对方释放锁，导致永久阻塞。

### 示例

事务1：

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- 锁住行1
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 等待行2
```

事务2：

```sql
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;   -- 锁住行2
UPDATE accounts SET balance = balance + 50 WHERE id = 1;   -- 等待行1
```

此时死锁发生。

### PostgreSQL 的死锁检测

- 后台进程 **deadlock_timeout**（默认 1 秒）会检测循环等待。
- 一旦发现，PostgreSQL 会回滚**其中一个事务**（代价最小的事务），并抛出错误：
  ```
  ERROR: deadlock detected
  HINT: See server log for query details.
  ```
- 应用层应该捕获此错误并重试整个事务。

**避免死锁的方法**：

- 以固定顺序访问资源（例如总是先更新 `id=1` 再更新 `id=2`）。
- 避免在事务中等待用户输入。
- 减少事务长度，尽快提交。
- 适当使用 `SELECT FOR UPDATE` 提前锁住需要的行。

## 五、锁的监控与管理

### 1. 查看当前锁等待

```sql
SELECT
    blocked_locks.pid AS blocked_pid,
    blocking_locks.pid AS blocking_pid,
    blocked_activity.query AS blocked_query,
    blocking_activity.query AS blocking_query
FROM pg_locks blocked_locks
JOIN pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_locks blocking_locks ON
    blocked_locks.locktype = blocking_locks.locktype AND
    blocked_locks.relation = blocking_locks.relation AND
    blocked_locks.page = blocking_locks.page AND
    blocked_locks.tuple = blocking_locks.tuple
JOIN pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;
```

### 2. 强制终止阻塞的进程（谨慎使用）

```sql
SELECT pg_terminate_backend(pid_of_blocker);
```

### 3. 查看数据库中的锁汇总（扩展视图）

```sql
SELECT locktype, relation::regclass, mode, granted, count(*)
FROM pg_locks
GROUP BY 1,2,3,4;
```

## 六、常见误区与最佳实践

| 误区                 | 真相                                                              |
| -------------------- | ----------------------------------------------------------------- |
| `SELECT` 会锁住行    | 普通 `SELECT` 不加行锁，MVCC 保证一致性视图。                     |
| 主键冲突不会引发锁   | 主键冲突会短暂地加行锁检查唯一性，但不持久。                      |
| 外键不会加锁         | 外键约束检查会加 `FOR KEY SHARE` 锁，防止被引用行删除或修改键列。 |
| 死锁只出现在复杂 SQL | 即使简单并发 `UPDATE` 也可能死锁（如互相等待对方更新不同行）。    |
| 所有锁都可以看到     | 某些轻量级锁（SpinLock）不暴露在 `pg_locks` 中。                  |

### 最佳实践

1. **优先使用 MVCC**，不要轻易加显式锁。“优先使用 MVCC” = 平时放心用普通 SELECT，让数据库自动管理版本；仅在写写冲突无法通过乐观重试解决时，才考虑显式行锁或表锁。这样做能获得最高的读写并发性能，同时避免不必要的锁等待。
2. **事务尽可能短**，减少锁持有时间。
3. **按相同顺序访问表/行**，避免死锁。
4. **使用 `SELECT FOR UPDATE` 时注意目的**：仅在需要防止其他事务修改时才用。
5. **监控长事务**，它们会堆积锁并阻碍清理（VACUUM）。
6. **避免在应用程序中使用 `LOCK TABLE`**，除非你确切知道后果。

## CTE 与视图的对比详解（以 PostgreSQL 为例）

**CTE（Common Table Expression，公用表表达式）** 和 **视图（View）** 都是 SQL 中用于封装查询逻辑的方式，但它们在本质、用途和性能上有显著区别。

---

## 一、核心定义

| 特性         | CTE                                                             | 视图                                                                               |
| ------------ | --------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| **本质**     | 一个**临时的命名查询结果集**，只存在于单个 SQL 语句的执行期间。 | 一个**存储的查询定义**，作为数据库对象持久存在，可以被多个查询重复使用。           |
| **生命周期** | 仅在当前 SQL 语句执行时存在。                                   | 永久存在于数据库中，除非被显式删除。                                               |
| **别名**     | 用 `WITH` 子句定义，像一个临时的表或视图。                      | 用 `CREATE VIEW` 定义，像一个虚拟表。                                              |
| **递归能力** | **支持递归**（`WITH RECURSIVE`），可处理树形或图形数据。        | 不支持递归（标准 SQL 中不行，某些数据库支持递归视图但少见）。                      |
| **参数化**   | 可以接受外部参数（通过查询中的变量）。                          | 普通视图不能参数化，但可以创建参数化表值函数（如 PostgreSQL 的 `RETURNS TABLE`）。 |

---

## 二、基本语法示例

### CTE

```sql
-- 简单的 CTE
WITH recent_orders AS (
    SELECT * FROM orders WHERE order_date > '2026-01-01'
)
SELECT user_id, COUNT(*)
FROM recent_orders
GROUP BY user_id;

-- 递归 CTE（查询组织层级）
WITH RECURSIVE employee_tree AS (
    SELECT id, name, manager_id, 1 AS level
    FROM employees WHERE manager_id IS NULL
    UNION ALL
    SELECT e.id, e.name, e.manager_id, et.level + 1
    FROM employees e
    JOIN employee_tree et ON e.manager_id = et.id
)
SELECT * FROM employee_tree;
```

### 视图

```sql
-- 创建视图
CREATE VIEW recent_orders_view AS
SELECT * FROM orders WHERE order_date > '2026-01-01';

-- 使用视图（可以像表一样查询）
SELECT user_id, COUNT(*)
FROM recent_orders_view
GROUP BY user_id;

-- 删除视图
DROP VIEW recent_orders_view;
```

---

## 三、主要区别

| 维度         | CTE                                                                                                                                                                                                    | 视图                                                                         |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| **持久性**   | 仅存在于当前查询。                                                                                                                                                                                     | 持久存储，可被所有会话、所有查询使用。                                       |
| **重用性**   | 不能跨查询重用（除非写为宏）。                                                                                                                                                                         | 可被任何查询引用，简化复杂 SQL。                                             |
| **性能优化** | 默认是优化器的“优化隔离墙”（PostgreSQL 中优化器通常不会将 CTE 内部下推或合并，除非使用 `MATERIALIZED` / `NOT MATERIALIZED` 控制）。但 PostgreSQL 12+ 允许 CTE 被内联（如果只被引用一次，且无副作用）。 | 视图通常会被展开为原始 SQL，优化器可以应用所有规则（如谓词下推、索引选择）。 |
| **副作用**   | 如果 CTE 包含 `INSERT`/`UPDATE`/`DELETE`（数据修改 CTE），则会产生实际影响。                                                                                                                           | 普通视图只读（除非是可更新视图或带 `INSTEAD OF` 触发器）。                   |
| **嵌套**     | CTE 可以引用前一个 CTE（线性引用），但不能自引用（除了递归）。                                                                                                                                         | 视图可以引用其他视图，但深度过深可能影响性能。                               |
| **调试**     | 便于逐步拆解复杂查询，每个 CTE 可单独测试。                                                                                                                                                            | 便于重用，但可能掩盖底层表结构变化。                                         |

---

## 四、性能方面（PostgreSQL 特性）

### CTE 的行为

在 PostgreSQL 中，CTE 默认是 **物化的优化边界**：

- 优化器不会将外部条件（如 `WHERE`）**下推** 到 CTE 内部，也不会将 CTE 与外部表合并执行。
- 这可能导致**性能陷阱**：例如 CTE 返回了 100 万行，外部只取 10 行，但 CTE 仍计算了全部 100 万行。
- PostgreSQL 12+ 引入 `MATERIALIZED` / `NOT MATERIALIZED` 提示：
  ```sql
  WITH t AS NOT MATERIALIZED (SELECT ...) SELECT ...;  -- 将 CTE 视为子查询展开
  WITH t AS MATERIALIZED (SELECT ...) SELECT ...;      -- 强制物化
  ```

### 视图的行为

- 视图被**展开**为子查询，优化器可以应用**谓词下推**、**列裁剪** 等。
- 对于复杂嵌套视图，展开后可能导致巨大 SQL，优化器也可能吃力。
- 物化视图（`CREATE MATERIALIZED VIEW`）则完全不同：它是物理存储的表，可建立索引，但需要手动刷新。

## 五、典型使用场景

| 场景                                           | 推荐                              | 原因                                                             |
| ---------------------------------------------- | --------------------------------- | ---------------------------------------------------------------- |
| 单次查询中需要多次引用同一个子查询结果         | **CTE**                           | 避免重复编写子查询，且可物化一次（`MATERIALIZED`）避免重复计算。 |
| 递归查询（如组织关系、树形结构）               | **CTE**                           | 视图不支持递归（标准 SQL）。                                     |
| 将复杂逻辑封装为可复用的“虚拟表”               | **视图**                          | 多个查询、多个应用可以共用该视图。                               |
| 对查询进行分步调试或增强可读性                 | **CTE**                           | 可以在一个语句内分段，临时性强。                                 |
| 需要给某些用户提供预定义的数据范围（行级安全） | **视图**                          | 视图可以限定行或列，实现安全隔离。                               |
| 需要参数化查询（传入外部值）                   | **视图** 不直接支持；可使用表函数 | CTE 可以直接在 `WHERE` 中使用参数（如 `WHERE id = $1`）。        |
| 数据修改（如先删除再插入的原子操作）           | **CTE**（数据修改 CTE）           | 可将多个 DML 放在一个 CTE 网络中，原子执行。                     |

## 六、实际案例比较

### 1. 查询每个部门工资最高的员工（使用 CTE）

```sql
WITH dept_max AS (
    SELECT dept_id, MAX(salary) AS max_salary
    FROM employees GROUP BY dept_id
)
SELECT e.*
FROM employees e
JOIN dept_max d ON e.dept_id = d.dept_id AND e.salary = d.max_salary;
```

这个 CTE 只用于这一个查询，不需要保存到数据库。

### 2. 同样的需求使用视图

```sql
CREATE VIEW dept_max_salary AS
SELECT dept_id, MAX(salary) AS max_salary
FROM employees GROUP BY dept_id;

-- 之后可以在任何地方使用该视图
SELECT e.*
FROM employees e
JOIN dept_max_salary d ON e.dept_id = d.dept_id AND e.salary = d.max_salary;
```

视图永久存在，可被多个查询共享。

## 七、总结表格

|                | CTE                                            | 视图                               |
| -------------- | ---------------------------------------------- | ---------------------------------- |
| **持久化**     | 否                                             | 是                                 |
| **作用域**     | 单个 SQL 语句                                  | 整个数据库                         |
| **可重用**     | 仅当前语句内                                   | 跨会话、跨查询                     |
| **递归支持**   | 是（`WITH RECURSIVE`）                         | 否（标准 SQL）                     |
| **数据修改**   | 支持（带有 `INSERT`/`UPDATE`/`DELETE` 的 CTE） | 普通视图只读（特殊可更新视图除外） |
| **优化器行为** | 默认优化边界（可控制物化）                     | 展开后优化                         |
| **使用场景**   | 单次复杂查询、递归、可读性                     | 封装通用逻辑、安全、复用           |

---

## 八、常见误区

- **误区1**：“CTE 一定比子查询慢” —— 不一定，在 PostgreSQL 中 CTE 可以物化，避免重复计算；而子查询可能多次执行。
- **误区2**：“视图没有性能开销” —— 视图的展开可能会阻止某些优化（如索引扫描），但通常影响不大。
- **误区3**：“CTE 就是临时表” —— 更像“可命名的子查询”，且不一定物化为磁盘临时表。

---

**一句话建议**：

- 需要**递归**、**单次复杂查询**、**临时分步调试** → 用 CTE。
- 需要**长期重用**、**安全封装**、**提供给不同应用** → 用视图。

在 PostgreSQL 中，**用户**和**角色**本质上是一样的——两者都被称为“角色”（Role）。只是习惯上：

- 把拥有 **LOGIN** 权限的角色称为“用户”。
- 把没有 LOGIN 权限、仅用于管理权限集合的角色仍称为“角色”。

PostgreSQL 用统一的 **角色** 模型来管理数据库访问权限，简化了权限体系。

## 1. 核心概念

- **角色（Role）**：可以拥有数据库对象（如表、视图）并具有某些系统权限（如创建数据库、复制流等）。角色可以是一个用户，也可以是一个“组角色”（其他角色的容器）。
- **用户（User）**：只是 `LOGIN` 属性为 `TRUE` 的角色。能够连接到数据库、执行查询。
- **组角色（Group Role）**：`LOGIN` 属性为 `FALSE` 的角色，通常用来将一组权限集中赋予它，再把其他角色（用户或其他组）加入其中，实现权限继承。

> 在旧版 PostgreSQL（8.1 之前）中，`USER` 和 `GROUP` 是独立的命令，现在统一使用 `CREATE ROLE`，保留 `CREATE USER` 作为 `CREATE ROLE LOGIN` 的别名。

## 2. 创建与管理

### 创建角色

```sql
-- 创建一个普通角色（不能登录）
CREATE ROLE app_readonly;

-- 创建一个用户（可以登录）
CREATE USER alice WITH PASSWORD 'xxx';
-- 等价于：
CREATE ROLE alice WITH LOGIN PASSWORD 'xxx';
```

### 常见属性

- `LOGIN` / `NOLOGIN`：能否登录。
- `SUPERUSER` / `NOSUPERUSER`：是否数据库超级管理员。
- `CREATEDB`：能否创建数据库。
- `CREATEROLE`：能否创建/删除/修改其他角色。
- `INHERIT`：是否自动继承所属组角色的权限（默认 `INHERIT`）。
- `PASSWORD '...'`：设置密码。

### 修改角色

```sql
ALTER ROLE alice WITH CREATEDB;      -- 赋予创建数据库权限
ALTER ROLE alice WITH NOLOGIN;       -- 禁止登录（变成普通角色）
ALTER USER alice WITH PASSWORD 'new'; -- USER 和 ROLE 通用
```

### 删除角色

```sql
DROP ROLE alice;   -- 不能删除仍被依赖的对象所有者
DROP OWNED BY alice;  -- 先转移或删除该角色拥有的对象
```

## 3. 权限继承与成员关系

PostgreSQL 没有显式的“组对象”，而是通过 **角色成员关系** 实现权限复用。

```sql
-- 创建一个组角色（无登录）
CREATE ROLE sales_team NOLOGIN;

-- 给组角色赋予权限（例如对 sales 表的查询）
GRANT SELECT ON sales TO sales_team;

-- 将用户 alice 加入组
GRANT sales_team TO alice;
```

此时 `alice` 会**自动获得** `sales_team` 的所有权限（前提是 `alice` 的 `INHERIT` 属性为 `TRUE`，这是默认值）。

### 重要细节

- 如果以 `SET ROLE` 方式显式切换到组角色，可以暂时获得该角色的附加权限（如 `SET ROLE sales_team;`），但会失去当前用户的其他直接权限（取决于设置）。
- 成员关系可以多层嵌套：A 加入 B，B 加入 C → A 继承 B 和 C 的权限（默认情况下）。
- 使用 `WITH ADMIN OPTION` 可以让一个角色授予/撤销组内成员资格，类似于“组管理员”。

```sql
-- 让 alice 可以管理 sales_team 成员
GRANT sales_team TO alice WITH ADMIN OPTION;
-- 然后 alice 就可以执行：
GRANT sales_team TO bob;
```

## 4. 用户与角色的实际区别（DBA 视角）

| 特性             | 角色（Role）                   | 用户（User）                        |
| ---------------- | ------------------------------ | ----------------------------------- |
| 能否登录数据库   | 通常不能（`NOLOGIN`）          | 能（`LOGIN`）                       |
| 典型用途         | 权限集合、组                   | 人员、应用程序连接账户              |
| 能作为对象所有者 | 能                             | 能                                  |
| 能拥有密码       | 也能（但无意义，因为不能登录） | 通常设置密码                        |
| 创建命令         | `CREATE ROLE`                  | `CREATE USER` / `CREATE ROLE LOGIN` |

实际上，你完全可以把一个 **用户** 当作组来使用（赋予另一个用户成员关系），反之亦然。只是习惯上保持区分。

## 5. 查看角色信息

```sql
-- 列出所有角色
SELECT rolname, rolsuper, rolinherit, rolcreaterole, rolcreatedb, rolcanlogin
FROM pg_roles;

-- 查看角色成员关系
SELECT r.rolname AS role_name, m.rolname AS member_name
FROM pg_auth_members am
JOIN pg_roles r ON am.roleid = r.oid
JOIN pg_roles m ON am.member = m.oid;
```

## 6. 常见最佳实践

- **不要直接给用户授予对象级权限**（如 `SELECT ON table`）。改为创建组角色，将权限赋予组，再将用户加入组。这样未来修改权限只需要改组，无需逐个调整用户。
- 遵循最小权限原则：用户只拥有完成工作所需的最小权限。
- 使用 `SET ROLE` 临时提升权限执行特定任务，避免长期持有高权限。
- 对于应用程序，创建一个或多个专用登录角色，不要使用超级用户或数据库创建者。

## 总结

PostgreSQL 中 **用户 = 拥有 LOGIN 属性的角色**。角色体系统一管理登录、权限继承和成员关系，比传统的“用户/组”分离模型更灵活。正确使用角色成员关系和 `INHERIT` 属性，可以构建清晰、易维护的数据库权限体系。

- **角色** = PostgreSQL 里的统一概念。
- **用户** = 带有 `LOGIN` 权限的角色（可以登录数据库）。
- 没有 `LOGIN` 权限的角色，通常用来当**组**，方便一次把权限赋给多个用户。

一句话：**用户是能登录的角色，角色是权限的容器**。

### 补充 如何给与权限

给角色（role）赋予增删改查权限，使用 `GRANT` 命令：

### 1. 针对单个表

```sql
-- 赋予查询、插入、更新、删除四种权限
GRANT SELECT, INSERT, UPDATE, DELETE ON 表名 TO 角色名;

-- 或者用 ALL PRIVILEGES（等效于所有权限）
GRANT ALL PRIVILEGES ON 表名 TO 角色名;
```

### 2. 针对某个 schema 下的所有现有表

```sql
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA schema名 TO 角色名;
```

### 3. 顺便赋予 schema 的访问权限（必须，否则无法看到表）

```sql
GRANT USAGE ON SCHEMA schema名 TO 角色名;
```

### 示例

```sql
-- 创建角色
CREATE ROLE app_user;

-- 给角色赋予 sales 表的增删改查
GRANT SELECT, INSERT, UPDATE, DELETE ON sales TO app_user;

-- 同时允许访问 public schema
GRANT USAGE ON SCHEMA public TO app_user;

-- 设定默认权限：以后在 public schema 中新建的表，自动给 app_user 增删改查权限
ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;
```

之后把其他用户加入该角色，他们就能自动获得这些权限。

## 数据备份与恢复

PostgreSQL的备份与恢复主要分为两大类：**逻辑备份**与**物理备份**。

### 💾 一、逻辑备份：单库级，跨版本神器

逻辑备份是通过 `pg_dump` 等工具，将数据库结构（模式、表等）和数据导出为SQL脚本或归档文件。这种方式的优点是**跨版本、跨平台**，你甚至可以在恢复时只选择特定的表。

- **备份（pg_dump）**：导出整个数据库或指定的表。推荐使用 `-F c` 自定义格式，它支持压缩和并行恢复，功能更强大。
- **恢复（pg_restore 与 psql）**：
  - 对于自定义格式的备份文件 (`.dump`)，使用 `pg_restore` 恢复。它非常灵活，可以让你只恢复特定的表 (`-t`) 或模式 (`-n`)。
  - 对于SQL脚本文件 (`.sql`)，通常使用 `psql` 命令行工具来执行它。

### 💾 二、物理备份：整库级，高性能恢复

物理备份直接复制数据库的底层数据文件，备份和恢复速度通常比逻辑备份更快。它特别适合**大型数据库**，并能配合WAL日志实现**时间点恢复（PITR）**。

- **备份（pg_basebackup）**：这是PostgreSQL自带的物理热备工具，可以在数据库运行时进行全量备份。它会在备份开始时执行一个检查点，并将内存中的脏数据刷新到磁盘，确保备份数据的一致性。
- **恢复**：使用 `pg_basebackup` 备份的文件进行恢复时，通常需要先停止数据库服务，将备份文件解压回数据目录，然后启动数据库完成恢复。

### ⏳ 三、时间点恢复（PITR）：数据找回"后悔药"

PITR（Point-In-Time Recovery）是物理备份的一项重要功能，允许你将数据库恢复到**过去的任意一个时间点**。它的实现原理类似于一个持续录制的“存档”：`pg_basebackup` 提供一个全量备份作为“基础镜像”，而连续的**WAL（Write-Ahead Log）** 文件（即预写式日志）则记录了之后的所有数据变更。恢复时，系统会从全量备份开始，然后“重放”WAL日志中的操作，直到你指定的目标时刻为止。

### 🛠️ 四、主要工具与策略对比

| 特性                  | `pg_dump` / `pg_restore` (逻辑备份)        | `pg_basebackup` + WAL (物理备份)                       |
| :-------------------- | :----------------------------------------- | :----------------------------------------------------- |
| **备份粒度**          | 单个数据库或表                             | 整个PostgreSQL实例                                     |
| **恢复粒度**          | 可恢复整个库或部分表                       | 一般需恢复整个实例                                     |
| **跨版本兼容性**      | **优秀**，可跨大版本、不同架构恢复         | **较差**，依赖特定的PostgreSQL主版本                   |
| **恢复速度**          | 较慢，适用于中小型数据库                   | **较快**，直接拷贝文件，适合大型数据库                 |
| **时间点恢复 (PITR)** | 不支持                                     | **支持**，可恢复到任意时刻                             |
| **日常操作影响**      | 影响小，在线备份                           | 影响小，在线热备                                       |
| **主要应用场景**      | **数据迁移**、**选择性恢复**、开发测试环境 | **生产环境**、**灾难恢复 (DR)**、**时间点恢复 (PITR)** |

#### 高端场景：第三方工具

对于需要**自动化、集中管理和增量备份**的企业级生产环境，建议评估专业的第三方工具，如 `pgBackRest` 或 `Barman`。它们能显著简化备份流程并增强可靠性。

### 🎯 五、选型建议：我该用哪种？

- **开发/测试/小库**：首选 `pg_dump`。它简单直接，非常适合配合版本控制管理数据库结构。
- **生产环境/大库/灾备**：采用 **`pg_basebackup` + WAL 归档**的组合策略。这是PostgreSQL推荐的标准生产环境备份方案。
- **数据迁移**：使用 **`pg_dump`** 逻辑备份。它是跨大版本、跨平台（如从Windows迁移到Linux）最稳妥的选择。
- **误删/误更新**：依赖 **WAL 归档**进行PITR（时间点恢复），这是唯一能帮你找回过去某一时刻数据的方法。

#### 生产环境最佳实践参数

在 `postgresql.conf` 中，至少要设置以下参数，这是开启物理备份和PITR能力的基础：

```ini
wal_level = replica          # 确保记录足够的信息用于复制或PITR
archive_mode = on            # 开启归档模式
archive_command = 'cp %p /path/to/archive/%f'  # 将WAL日志文件复制到归档目录
```

此外，**务必定期执行恢复演练**，确保备份文件是可用的。

下面用具体命令示例说明 PostgreSQL 的备份与恢复。

## 1. 逻辑备份（pg_dump + pg_restore）

**场景**：备份 `mydb` 数据库，稍后恢复到新库 `newdb`。

### 备份

```bash
# 导出为自定义格式（压缩、可并行恢复）
pg_dump -F c -d mydb -U postgres -f mydb.dump
```

### 恢复

```bash
# 先创建空数据库
createdb -U postgres newdb

# 恢复
pg_restore -U postgres -d newdb mydb.dump
```

恢复时可以只恢复特定表：`pg_restore -t employees -d newdb mydb.dump`

---

## 2. 物理备份（pg_basebackup）

**场景**：全量备份整个 PostgreSQL 实例，用于灾难恢复。

```bash
# 进行基础备份，输出到 /backup/base 目录，启用压缩和进度显示
pg_basebackup -D /backup/base -Fp -P -U replicator -h localhost
```

- `-D`：备份目标目录
- `-Fp`：输出为普通文件（plain format）
- `-P`：显示进度
- 需要配置 `replica` 用户有 `REPLICATION` 权限，并在 `postgresql.conf` 中开启 `wal_level = replica`

恢复时：停止数据库，清空数据目录，将备份文件复制回去，然后重启。

---

## 3. 时间点恢复（PITR）示例

**前提**：已启用 WAL 归档（`archive_mode=on`，`archive_command` 将 WAL 复制到归档目录）。

**场景**：误删数据发生在 `2025-05-10 10:30:00`，想恢复到之前一刻。

### 步骤

1. 恢复物理备份到数据目录。
2. 在数据目录创建 `recovery.signal` 文件（PG 12+）。
3. 在 `postgresql.conf` 中添加恢复目标：
   ```ini
   restore_command = 'cp /archive/%f %p'   # 从归档目录拉取 WAL
   recovery_target_time = '2025-05-10 10:29:55'
   recovery_target_action = 'promote'
   ```
4. 启动数据库，它会自动应用 WAL 直到指定时间点，然后提升为正常模式。

结果：数据库回到 10:29:55 的状态，误删除的数据被找回。

## pgsql 中的扩展系统

PostgreSQL 的扩展（Extension）是一个将多个数据库对象（如函数、数据类型、索引方法等）打包，然后方便安装、更新和移除的框架。

### 扩展的"三驾马车"

一个扩展通常由以下三部分组成：

- 控制文件：扩展的“身份证”，记录名称、版本等基本信息。

- SQL 脚本文件：具体的“安装”脚本，包含了创建各类数据库对象的 SQL 命令。

- 共享库文件：可选的部分，通常由 C 语言编写，为实现高性能功能提供的底层代码。

你只需要一个`CREATE EXTENSION`命令，PostgreSQL 就会根据这些定义好的文件，自动完成所有对象的创建和安装。

### 🛠️ 扩展的生命周期管理

扩展的管理非常简单直观，主要围绕以下三个命令展开：

| 命令                   | 作用                                                                             | 示例                                          |
| :--------------------- | :------------------------------------------------------------------------------- | :-------------------------------------------- |
| **`CREATE EXTENSION`** | 安装并启用一个扩展。如果加上 `IF NOT EXISTS` 子句，可以防止重复安装。            | `CREATE EXTENSION IF NOT EXISTS "uuid-ossp";` |
| **`ALTER EXTENSION`**  | 主要用于将扩展更新到新版本，例如修复 Bug 或增加新功能。                          | `ALTER EXTENSION postgis UPDATE TO '3.1.0';`  |
| **`DROP EXTENSION`**   | 卸载并删除一个扩展。这是扩展机制的优势所在，它会自动清理掉该扩展创建的所有对象。 | `DROP EXTENSION "uuid-ossp";`                 |

**一个重要的概念：先安装包，后创建扩展**
整个流程分为两步：

1.  **系统级安装**：你需要先在操作系统层面安装扩展的软件包。在 Ubuntu/Debian 系统上，通常使用 `apt-get install` 命令。
2.  **数据库级创建**：安装好软件包后，再连接到具体的数据库，执行 `CREATE EXTENSION` 命令来完成启用。

另外，像 `postgis` 或 `pg_stat_statements` 这类通过 `shared_preload_libraries` 参数来加载共享库的高级扩展，安装后需要**重启数据库服务**才能生效。

### 🧩 热门扩展举例

PostgreSQL 拥有一个非常丰富的扩展生态，以下是一些在不同领域应用广泛的扩展：

- **通用功能**
  - **`uuid-ossp`**：生成全球唯一的 UUID 标识符。
  - **`pgcrypto`**：提供哈希、加密和解密等密码学函数。
  - **`citext`**：提供大小写不敏感的文本类型，简化字符串查询。
  - **`hstore`**：在单列中存储键值对，适合存储半结构化数据。

- **性能与监控**
  - **`pg_stat_statements`**：收集和统计 SQL 语句的执行性能，是 DBA 进行SQL优化的核心工具。
  - **`pg_partman`**：自动化管理分区表的创建和维护，简化分区方案。

- **专业领域**
  - **`PostGIS`**：地理空间数据库扩展，添加对地理对象的支持，是 GIS 系统的首选数据库。
  - **`TimescaleDB`**：专为时间序列数据（如 IoT、监控数据）优化，提供高效的存储和查询。
  - **`pg_trgm`**：基于三元模型提供模糊匹配和全文搜索能力，常用于实现“搜索建议”功能。

### 💡 如何探索更多扩展

想了解还有哪些可用扩展，可以试试以下方法：

- **查看已安装扩展**：登录数据库后，使用 `\dx` 元命令可以列出当前数据库中所有已安装的扩展及其版本。
- **搜索更多扩展**：可以访问在线扩展目录（如 [https://ext.pgsty.com](https://ext.pgsty.com)）来浏览更详尽的扩展列表。
- **检查官方文档**：PostgreSQL 的 `contrib` 模块是官方提供的扩展集合，通常随主安装包一同发布，文档中有详细说明。

### ⚠️ 安装权限说明

通常只有 **超级用户（Superuser）** 才有权限执行 `CREATE EXTENSION` 命令。不过，如果一个扩展在其控制文件中被标记为 `trusted`（受信任的），那么普通用户也可以安装它，前提是拥有当前数据库的 `CREATE` 权限。

### 🧩 扩展与模块的区别

- **扩展**：通常是标准化的内容，用于在数据库中**创建新的对象**，会被 `pg_dump` 工具识别并处理。
- **模块**：通常指**不包含SQL对象**的独立共享库。它们通常不创建新对象，而是在后台运行以改变数据库行为，需要使用 `shared_preload_libraries` 加载。

PostgreSQL 内置**全文搜索**（Full-Text Search），无需额外安装扩展，适合在大量文本中快速查找关键词，支持模糊匹配、词干处理、排序等。

## 核心概念

- **`tsvector`**：把文本拆解成“词素”的排序列表（已去停用词、归一化）。
- **`tsquery`**：要搜索的条件，支持 `&`(与)、`|`(或)、`!`(非) 组合。
- **`@@`**：匹配操作符，检查 tsvector 是否满足 tsquery。

## 简单示例

```sql
-- 1. 建表并插入数据
CREATE TABLE articles (id int, title text, body text);
INSERT INTO articles VALUES (1, 'PostgreSQL full-text search', 'Full text search is fast and flexible.');

-- 2. 查询 body 字段包含 'fast' 或 'flexible' 的记录
SELECT * FROM articles
WHERE to_tsvector('english', body) @@ to_tsquery('english', 'fast | flexible');
```

## 常用函数

| 函数                            | 作用                                   |
| ------------------------------- | -------------------------------------- |
| `to_tsvector(config, text)`     | 文本转 tsvector                        |
| `to_tsquery(config, text)`      | 关键词转 tsquery                       |
| `plainto_tsquery(config, text)` | 把普通文字转成 tsquery（默认空格为 &） |
| `ts_rank(tsvector, tsquery)`    | 计算相关性分数                         |

## 加速：GIN 索引

```sql
-- 创建索引（一般基于 tsvector 列）
CREATE INDEX idx_articles_body ON articles USING GIN (to_tsvector('english', body));

-- 查询会自动使用索引
SELECT * FROM articles
WHERE to_tsvector('english', body) @@ to_tsquery('english', 'fast');
```

## 整列预生成（推荐）

```sql
-- 添加一个 tsvector 列并自动更新
ALTER TABLE articles ADD COLUMN text_vec tsvector
GENERATED ALWAYS AS (to_tsvector('english', coalesce(title,'') || ' ' || coalesce(body,''))) STORED;

CREATE INDEX ON articles USING GIN (text_vec);

-- 查询直接使用该列
SELECT * FROM articles WHERE text_vec @@ to_tsquery('english', 'fast');
```

## 配置语言

不同的语言有不同的分词规则，例如 `'russian'`、`'simple'`（不分词）。中文需要额外的扩展如 `zhparser`。

## 与 `LIKE` / `ILIKE` 对比

- `LIKE '%word%'`：不能利用索引（除非使用 `pg_trgm`），不支持词干（“runs”搜不到“run”）。
- 全文搜索：速度快、支持权重、排名、近义词等。

一句话：**全文搜索 = `to_tsvector` + `to_tsquery` + `@@` + GIN 索引**，适合搜索文章、日志等长文本。

`pgvector` 是一个开源的 PostgreSQL 扩展，它为这个强大的关系型数据库添加了存储、查询和索引高维向量的能力。简单来说，它让 PostgreSQL 摇身一变，也拥有了向量数据库的核心功能。

它的优势在于与 PostgreSQL 生态的无缝集成，让你能在一个数据库中同时处理传统的结构化数据和 AI 应用中常见的向量数据，并享受 `JOIN`、事务（ACID）和时间点恢复（PITR）等全部功能。它已成为构建 RAG（检索增强生成）应用的基石。

### 📊 核心概念：向量数据类型与距离函数

1.  **数据类型**：`pgvector` 通过 `vector` 数据类型来存储高维向量。你需要**预先指定向量的维度**。

    ```sql
    -- 创建一个用于存储 3 维向量的表
    CREATE TABLE items (id bigserial PRIMARY KEY, embedding vector(3));
    ```

    除 `vector` 外，`pgvector` 还支持其他类型以满足不同需求：
    | 类型 | 说明 | 用途 |
    | :--- | :--- | :--- |
    | **`halfvec`** | 半精度浮点数向量 | 在精度要求不高的场景节省存储空间 |
    | **`bit`** | 二进制向量 | 适用于哈希等二进制嵌入 |
    | **`sparsevec`** | 稀疏向量 | 高效存储绝大多数元素为零的向量 |

2.  **距离与相似度运算符**：这是向量搜索的核心，用于衡量向量间的相似程度。具体选哪个，取决于你的嵌入模型训练时用的什么。
    | 运算符 | 含义 | 说明 |
    | :--- | :--- | :--- |
    | `<->` | **欧几里得距离** (L2) | 计算两点间直线距离，值越小越相似 |
    | `<=>` | **余弦距离** | 关注方向而非大小，值越小越相似。其值 `= 1 - 余弦相似度`。 |
    | `<#>` | **负内积** (Negative Dot Product) | 返回负值，使得 `ORDER BY` 升序时能优先得到最相似的结果。当向量已归一化时，内积等价于余弦相似度。 |

### ⚙️ 关键技术详解

#### 1. 索引技术：HNSW vs. IVFFlat

当数据量变大时，**精确搜索** (暴力扫描所有向量) 会变得非常慢。`pgvector` 提供了两种**近似最近邻（ANN）** 索引，通过牺牲微小的精确度（召回率）来换取巨大的速度提升。

| 特性         | **HNSW** (Hierarchical Navigable Small World)            | **IVFFlat** (Inverted File with Flat Compression)        |
| :----------- | :------------------------------------------------------- | :------------------------------------------------------- |
| **原理**     | 构建多层图结构，如同在高速公路网络中导航                 | 通过聚类将向量分组，查询时只搜索最相关的几个组           |
| **性能**     | **查询更快，召回率更高**                                 | **构建更快，内存占用更低**                               |
| **适用场景** | **生产环境和多数场景的首选**，特别是**数据会持续插入**时 | **数据静态、有限的内存资源**或**索引需要频繁重建**的场景 |
| **数据要求** | 可以在空表上直接创建                                     | **必须在表中有数据之后**创建，以确保聚类效果             |
| **索引上限** | 最多支持 **2,000 维** 的向量                             | 最多支持 **2,000 维** 的向量                             |
| **关键参数** | `m`, `ef_construction`, `ef_search`                      | `lists`, `probes`                                        |

#### 2. 索引参数调优指南

为了帮你更直观地选择索引和参数，可以参考下表：

| 参数              | 索引类型 | 描述与推荐值                                                | 影响                                                                        |
| :---------------- | :------- | :---------------------------------------------------------- | :-------------------------------------------------------------------------- |
| `lists`           | IVFFlat  | 聚类数。公式: `sqrt(行数)`。例：100万行 -> `lists = 1000`。 | **值越大**：查询更快（搜索的列表少），**但召回率可能下降**。                |
| `probes`          | IVFFlat  | 查询时探测的列表数，默认1。公式: `sqrt(lists)`。            | **值越大**：召回率越高，**但查询速度越慢**。设为 `lists` 即退化为精确搜索。 |
| `m`               | HNSW     | 每层最大连接数，默认16，范围8-64。                          | **值越大**：召回率越高、构建越慢，**内存占用越高**。                        |
| `ef_construction` | HNSW     | 构建时动态候选列表大小，默认64，范围64-200。                | **值越大**：图质量越高、构建越慢，**召回率越高**。                          |
| `ef_search`       | HNSW     | 查询时动态候选列表大小，默认40。                            | **值越大**：召回率越高，**但查询速度越慢**。                                |

- **HNSW 索引示例**：
  ```sql
  -- 为余弦距离创建 HNSW 索引
  CREATE INDEX ON items USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
  ```
- **IVFFlat 索引示例**：
  ```sql
  -- 为余弦距离创建 IVFFlat 索引，并设置 100 个列表
  CREATE INDEX ON items USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
  ```
- **查询时调整参数**：
  ```sql
  SET hnsw.ef_search = 200;  -- 为当前会话提高 HNSW 查询召回率
  SET ivfflat.probes = 10;   -- 为当前会话提高 IVFFlat 查询召回率
  ```

### 🚀 实战：从嵌入向量到混合搜索

#### 1. 存储嵌入向量

假设你为商品生成了 3 维的文本嵌入向量，可以这样插入数据。

```sql
INSERT INTO items (embedding) VALUES ('[1,2,3]'), ('[4,5,6]');
```

#### 2. 向量相似度搜索

找到与目标向量 `[3,1,2]` 最相似的 5 个商品。

```sql
SELECT * FROM items
ORDER BY embedding <-> '[3,1,2]'
LIMIT 5;
```

将 `<->` 替换为 `<=>` 或 `<#>` 即可使用其他距离算法。

#### 3. 混合搜索

这是 `pgvector` 的核心优势：在一次查询中，将结构化条件过滤和向量相似度搜索结合起来。

```sql
-- 查找 2024 年后发布、且与查询向量最相似的 10 篇新闻
SELECT *
FROM news
WHERE publish_date > '2024-01-01' AND category = 'technology'
ORDER BY embedding <=> '[0.1, 0.2, ...]'
LIMIT 10;
```

### 💎 总结

总的来说，`pgvector` 不是要取代独立的向量数据库，而是为绝大多数的 PostgreSQL 使用者提供了一个**轻量、易集成**的解决方案。它极大地降低了 AI 应用（特别是 RAG）的架构复杂度和运维成本，让开发者可以在熟悉的 SQL 环境中，以极低的心智负担，优雅地解决“语义相似”搜索的问题，并利用 PostgreSQL 生态中丰富的工具来管理、迁移和备份你的数据。

## pgvector 与 专业向量数据的对比

PostgreSQL 的 pgvector 与常见向量数据库在使用体验上，核心区别在于**“SQL”与“API”**。

简单来说，pgvector 就像是给你的数据库装上一个新功能，一切操作都在你熟悉的 SQL 环境里完成；而专用数据库则会引入一套新的 API 和概念，需要你去学习和集成。

下面通过对比 pgvector 和几个主流向量数据库 (Milvus, Qdrant, Weaviate, Pinecone) 的使用模式、代码示例，来直观感受它们的区别。

### 1. 🛠️ 安装与设置

- **pgvector**：直接在现有 PostgreSQL 实例中创建扩展，无需额外服务，对DBA友好。

  ```sql
  CREATE EXTENSION vector;
  ```

- **Milvus / Qdrant / Weaviate**：通常需要独立安装或通过 Docker 启动服务，会新增一个独立的技术组件和运维负担。例如，用 Python 客户端连接：
  ```python
  # Milvus 连接示例
  from pymilvus import connections
  connections.connect(host='localhost', port='19530')
  ```
  ```python
  # Qdrant 客户端示例
  from qdrant_client import QdrantClient
  client = QdrantClient(host='localhost', port=6333)
  ```
- **Pinecone**：纯托管服务，无需自己维护服务器，但代码中需配置 API Key。

### 2. 🏗️ 存储结构：Table vs. Collection/Index

- **pgvector**：使用最熟悉的 **SQL 表 (Table)** 结构。

  ```sql
  -- 创建包含向量列的表
  CREATE TABLE documents (
      id BIGSERIAL PRIMARY KEY,
      content TEXT,
      embedding vector(1536)  -- 指定向量维度
  );
  ```

- **专用数据库**：引入新的概念，如 **Collection (Milvus/Qdrant)** 或 **Index (Pinecone)**，你需要为其定义 schema。
  ```python
  # Qdrant 创建 Collection 示例
  from qdrant_client import models
  client.create_collection(
      collection_name="my_collection",
      vectors_config=models.VectorParams(size=1536, distance=models.Distance.COSINE),
  )
  ```

### 3. ➕ 数据操作 (CRUD)

- **pgvector**：使用标准的 **SQL 命令 (`INSERT`, `UPDATE`, `DELETE`)**，无需学习新语法。

  ```sql
  INSERT INTO documents (content, embedding) VALUES ('...', '[0.1, 0.2, ...]');
  ```

- **专用数据库**：通过特定的 **SDK 方法**，如 `insert` (Milvus) 或 `upsert` (Qdrant/Pinecone)。
  ```python
  # Pinecone Upsert 示例 (将向量 upsert 到索引中)
  index.upsert(vectors=[("id-1", [0.1, 0.2, ...])])
  ```

### 4. 🔍 查询与检索

这是两者使用差异最大的地方，pgvector 完全拥抱 SQL，而专用数据库则提供专门的查询 API。

**相似度查询 (Similarity Search)**

- **pgvector**: 使用 `ORDER BY ... <->` 子句，很像传统的坐标查询。
  ```sql
  SELECT content, 1 - (embedding <=> '[0.1, 0.2, ...]') AS similarity
  FROM documents
  ORDER BY embedding <=> '[0.1, 0.2, ...]'  -- 按欧氏距离排序
  LIMIT 5;
  ```
- **Qdrant**: 使用 `client.search` 方法。
  ```python
  client.search(
      collection_name="my_collection",
      query_vector=[0.1, 0.2, ...],
      limit=5
  )
  ```

**元数据过滤 (Metadata Filtering)**

- **pgvector**: 直接在 `WHERE` 子句中添加条件，结合 `JOIN` 能力强大。
  ```sql
  SELECT * FROM documents
  WHERE metadata->>'category' = 'technology'
  ORDER BY embedding <=> '[0.1, 0.2, ...]'
  LIMIT 10;
  ```
- **Qdrant**: 在 `search` 方法中传入 `query_filter` 参数，使用其特定的 `Filter` 对象语法。
  ```python
  from qdrant_client import models
  client.search(
      ...,
      query_filter=models.Filter(
          must=[models.FieldCondition(key="category", match=models.MatchValue(value="technology"))]
      )
  )
  ```

### 5. 🚀 高级功能对比

这是专用数据库构建护城河的地方。从实践看，选择它们往往意味着你不仅是想要向量搜索，而是需要其独有的高级功能。

| 功能特性                     | pgvector             | Milvus       | Qdrant           | Weaviate        | Pinecone      |
| :--------------------------- | :------------------- | :----------- | :--------------- | :-------------- | :------------ |
| **混合搜索 (Hybrid Search)** | 需手动组合 SQL       | ✅ 原生支持  | ✅ 原生支持      | ✅ 原生支持     | ✅ 原生支持   |
| **内置Embedding模型**        | ❌ 不支持            | ❌ 不支持    | ⚠️ 集成FastEmbed | ✅ 内置多种模型 | ✅ 服务端生成 |
| **GPU 加速**                 | ❌ 不支持            | ✅ 支持      | ✅ 支持          | ❌ 不支持       | ❌ 不支持     |
| **多租户/命名空间**          | 通过`schema`或表实现 | ✅ Partition | ✅ 支持          | ✅ 支持         | ✅ Namespace  |
| **批量向量检索**             | 需循环查询           | ✅ 支持      | ✅ 批量搜索      | ✅ 支持         | ✅ 支持       |
| **多向量/命名向量**          | 通过多列实现         | ✅ 支持      | ✅ 多向量支持    | ✅ 支持         | ❌ 不支持     |

### 💎 总结

总而言之，pgvector 和专用向量数据库在使用上的对比，可以看作是一场“**SQL 的便捷性**”与“**专业 API 的强大功能**”之间的权衡。pgvector 的最大优势在于其熟悉的 SQL 接口和与 PostgreSQL 生态的无缝集成，让开发、运维更简单。而专用数据库则通过提供专门的 API 和丰富的功能，在特定场景下具备更强的优势。

**给你几个直截了当的选型建议**：

- **优先选 pgvector**：如果你的团队对 PostgreSQL 非常熟悉，追求极简架构，并且数据量在百万级以内。
- **考虑专用向量数据库**：当你明确需要**混合搜索**、数据量已达**千万级别**，或希望利用**GPU加速**提升性能时。

另外需要说明的是，这是一个快速发展的领域，像 Milvus（2019年开源）、Qdrant（2021年开源）这类专用向量数据库的功能和API也在持续演进。我的建议是从简入手，先用 pgvector 快速跑通流程，等它真的成为瓶颈时，再考虑迁移到能力更强的专用数据库。

## fastapi 与 pgsql 结合

在 FastAPI 里和 PostgreSQL 打交道，核心思路可以总结成一句话：**在异步应用中，使用异步的驱动和连接池**。

PostgreSQL 官方推荐的异步驱动是 `asyncpg`。市面上主流的 Python ORM（对象关系映射）库和底层数据库工具，也都围绕它构建了高效的异步支持。主要有四种主流方案，你可以根据项目的复杂程度和团队习惯来权衡选择。

### ⚔️ 四种主流技术方案对比

| 方案           | 核心工具                                    | 异步驱动  | 适用场景                                                |
| :------------- | :------------------------------------------ | :-------- | :------------------------------------------------------ |
| **专业ORM**    | **SQLAlchemy 2.0+** <br>强大的数据映射器    | `asyncpg` | **大型、复杂项目**，对模型关系、数据完整性要求高        |
| **轻量ORM**    | **Tortoise ORM** <br>类Django语法，快速开发 | `asyncpg` | **新手友好、快速原型**，追求开发效率和简洁性            |
| **极简数据层** | **Databases** <br>简单的连接池和SQL执行     | `asyncpg` | **偏好写原生SQL**，但希望获得异步和连接池支持的小型项目 |
| **原始驱动**   | **Asyncpg** <br>高性能底层驱动              | -         | **高性能场景**，需要对SQL有极致控制，不介意直接处理连接 |

### ⚙️ 核心技术点：异步连接池 (Connection Pool)

无论选择哪种方案，底层的连接池都至关重要。它可以复用数据库连接，避免重复的连接开销，从而显著提升高并发下的性能。

**方案一：使用 `databases` 构建轻量级异步数据访问层**
这种方式不用 ORM，能让开发者用 `async/await` 直接执行原生 SQL，兼顾了性能与便捷。

```python
import os
from fastapi import FastAPI
from contextlib import asynccontextmanager
from databases import Database

DATABASE_URL = os.getenv("DATABASE_URL")  # 格式: postgresql://user:pass@localhost/db
database = Database(DATABASE_URL)

@asynccontextmanager
async def lifespan(app: FastAPI):
    await database.connect()        # 启动时连接
    yield
    await database.disconnect()     # 关闭时断开

app = FastAPI(lifespan=lifespan)

@app.get("/users/{user_id}")
async def read_user(user_id: int):
    query = "SELECT * FROM users WHERE id = :id"
    return await database.fetch_one(query, values={"id": user_id})
```

_(以上代码源自社区示例，展示了如何通过lifespan管理连接的生命周期。)_

**方案二：使用 SQLAlchemy 2.0 构建异步 ORM**
SQLAlchemy 2.0 是 FastAPI 官方文档推荐的组合，它提供了强大的 ORM 能力，但也需要更多的配置。

首先，创建异步引擎和会话工厂是配置核心。

```python
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker
from sqlalchemy.orm import DeclarativeBase

# 注意URL格式：postgresql+asyncpg
SQLALCHEMY_DATABASE_URL = "postgresql+asyncpg://user:pass@localhost/db"
engine = create_async_engine(SQLALCHEMY_DATABASE_URL, echo=True)
AsyncSessionLocal = async_sessionmaker(engine, expire_on_commit=False)
```

_(以上配置模型展示了如何使用 `asyncpg` 驱动创建异步引擎。)_

在定义模型和数据访问层时，可以像下面这样操作。

```python
# 模型定义
class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True, index=True)
    email = Column(String, unique=True, index=True)

# 在 FastAPI 路由中使用
async def get_db():
    async with AsyncSessionLocal() as session:
        yield session

@app.post("/users/")
async def create_user(user: UserCreate, db: AsyncSession = Depends(get_db)):
    db_user = User(email=user.email)
    db.add(db_user)
    await db.commit()
    await db.refresh(db_user)
    return db_user
```

### 🚀 高级场景：在 FastAPI 中集成 pgvector

结合我们之前讨论的 `pgvector`，你可以在同一套异步架构中无缝地为 AI 应用增加向量检索能力。这里以 SQLAlchemy 为例：

1.  **定义向量模型**：在你的 SQLAlchemy 模型中，使用 `Vector` 类型。

```python
# models.py
from pgvector.sqlalchemy import Vector
from sqlalchemy import Column, Integer, String

class Document(Base):
    __tablename__ = "documents"
    id = Column(Integer, primary_key=True, index=True)
    content = Column(String)
    embedding = Column(Vector(1536))  # 假设使用1536维的嵌入模型
```

2.  **存入向量**：在 API 路由中，生成嵌入后直接存入数据库。

```python
embedding = generate_embedding(content) # 假设这是个获取向量的函数
db_doc = Document(content=content, embedding=embedding)
db.add(db_doc)
await db.commit()
```

3.  **相似度查询**：使用向量距离运算符（如 `<->` 欧氏距离，或 `<=>` 余弦距离）进行排序查询。

```python
from pgvector.sqlalchemy import L2Distance

@app.get("/search")
async def search(q: str):
    query_embedding = generate_embedding(q)
    results = await db.execute(
        select(Document).order_by(L2Distance(Document.embedding, query_embedding)).limit(5)
    )
    return results.scalars().all()
```

_(以上代码模式展示了如何将 `pgvector` 集成到 SQLAlchemy 异步架构中。)_

| 场景                               | 推荐方案              |
| :--------------------------------- | :-------------------- |
| **团队有SQLAlchemy经验，项目复杂** | 选 **SQLAlchemy 2.0** |
| **追求快速开发和简单API**          | 选 **Tortoise ORM**   |
| **写原生SQL，不要ORM**             | 选 **Databases**      |

## embedding 函数

实现`generate_embedding`函数，核心是将文本转化为数值向量。至于如何选择，可以基于是否需要数据完全私有、响应速度、中文优化等标准。

**总的来说：追求极致便捷就用OpenAI API；涉及敏感数据或需要长文本处理，选BGE系列；喜欢轻量快速，则text2vec是不错的选择。**

下面我为你整理了几种主流方案，并附上详细的代码示例。

### 主流方案概览与选型指南

| 方案                                    | 核心优势                         | 主要考量                       | 适用场景                                      |
| :-------------------------------------- | :------------------------------- | :----------------------------- | :-------------------------------------------- |
| **1. 在线 API (如 OpenAI)**             | 质优、稳定、无需硬件             | 依赖外部服务、数据隐私、有成本 | 重视效果的非敏感业务、通用场景                |
| **2. 本地模型 - BGE系列**               | 中文SOTA、向量维度高、可私有部署 | 硬件要求高、推理速度相对较慢   | **中文密集型任务**，如智能客服、RAG、文本聚类 |
| **3. 本地模型 - text2vec**              | 中文优化、轻量快速、易上手       | 语义丰富度略弱于大型模型       | 资源有限、对响应速度要求高的中文场景          |
| **4. 本地模型 - Sentence-Transformers** | 国际化、通用性最强、多语言支持好 | 中文原生支持相对BGE较弱        | 通用文本向量化、英文或多语言应用              |

### 1. 在线 API (以 OpenAI 为例)

这种方式最直接，调用远程服务，省去了本地部署和维护的麻烦。

**代码示例：**

```python
from openai import AsyncOpenAI  # 异步客户端，更适合FastAPI
import os

# 初始化客户端
client = AsyncOpenAI(api_key=os.getenv("OPENAI_API_KEY"))

async def generate_embedding_async(content: str) -> list[float]:
    """异步使用 OpenAI 模型生成文本向量"""
    try:
        response = await client.embeddings.create(
            model="text-embedding-3-small",  # 高性价比模型
            input=content,
            # encoding_format="float"      # 可指定返回格式
        )
        return response.data[0].embedding
    except Exception as e:
        print(f"OpenAI API error: {e}")
        # 根据需求决定是重试还是返回空向量或抛出异常
        raise
```

> 此模式参考了 OpenAI 官方API的使用方式。LangChain 等框架也提供了类似封装。

### 2. BGE (BAAI General Embedding) 系列

由北京智源研究院推出的BGE系列，是公认的**中文语义向量模型中的佼佼者**。其对中文语义的理解深度，尤其适合构建各类中文AI应用。

#### 方案A：利用`FlagEmbedding`库 (最推荐)

这是官方推荐的简单用法。

```python
from FlagEmbedding import FlagModel
import torch

# 选择模型，这里是 BAAI/bge-large-zh-v1.5
model = FlagModel('BAAI/bge-large-zh-v1.5',
                  query_instruction_for_retrieval="为这个句子生成表示以用于检索相关文章：",
                  use_fp16=True)  # 使用半精度节省显存

def generate_embedding_bge(content: str) -> list[float]:
    """使用 FlagEmbedding 生成 BGE 向量"""
    embedding = model.encode(content)
    return embedding.tolist()
```

#### 方案B：使用`LangChain`集成

如果你的项目已经使用了`LangChain`，`HuggingFaceBgeEmbeddings`是最佳选择。

```python
from langchain_huggingface import HuggingFaceBgeEmbeddings

# 初始化模型
model_name = "BAAI/bge-large-zh-v1.5"
encode_kwargs = {'normalize_embeddings': True} # 归一化向量

bge_embeddings = HuggingFaceBgeEmbeddings(
    model_name=model_name,
    model_kwargs={'device': 'cuda'}, # 有GPU则用，无则 'cpu'
    encode_kwargs=encode_kwargs
)

# 生成嵌入
def generate_embedding_langchain(content: str):
    return bge_embeddings.embed_query(content) # 同步方法，返回 list[float]
```

#### 方案C：直接使用`Transformers`库 (最底层)

提供最大灵活性，但也最复杂。

```python
from transformers import AutoTokenizer, AutoModel
import torch

tokenizer = AutoTokenizer.from_pretrained('BAAI/bge-large-zh-v1.5')
model = AutoModel.from_pretrained('BAAI/bge-large-zh-v1.5')
model.eval()

def generate_embedding_transformers(content: str):
    encoded_input = tokenizer(content, padding=True, truncation=True, return_tensors='pt', max_length=512)
    with torch.no_grad():
        model_output = model(**encoded_input)
        sentence_embeddings = model_output.last_hidden_state[:, 0]  # 取 [CLS] token 向量
    return sentence_embeddings.squeeze().tolist()
```

### 3. `text2vec-base-chinese`

这是一个专门为中文优化的、**轻量且高效的句子嵌入模型**，非常适合资源有限或追求快速响应的应用。

**代码示例：**

```python
from text2vec import SentenceModel

# 加载模型
model = SentenceModel('shibing624/text2vec-base-chinese')

def generate_embedding_text2vec(content: str) -> list[float]:
    """使用 text2vec 生成 768 维中文向量"""
    embeddings = model.encode([content])  # 支持批量
    return embeddings[0].tolist()
```

> 官方说明指出，该模型为`shibing624/text2vec-base-chinese`，将句子映射到**768维的稠密向量空间**，适用于句子嵌入、文本匹配、语义搜索等任务。

### 4. `Sentence-Transformers` (通用模型)

这个库提供了海量的预训练模型，是进行通用文本嵌入的首选，尤其适合英文或多语言场景。

**代码示例：**

```python
from sentence_transformers import SentenceTransformer

# 选择模型，如 'all-MiniLM-L6-v2'（英文）或 'paraphrase-multilingual-MiniLM-L12-v2'（多语言）
model = SentenceTransformer('paraphrase-multilingual-MiniLM-L12-v2')

def generate_embedding_sbert(content: str) -> list[float]:
    """使用 Sentence-Transformers 生成向量"""
    embedding = model.encode(content, convert_to_numpy=True)
    return embedding.tolist()
```

> 示例展示了`SentenceTransformer`库的基础用法。更多预训练模型可在其官网查找。

### 💎 性能优化建议

- **选择合适的硬件**：`BGE-large`等大模型对GPU显存有要求。轻量的`text2vec`和`all-MiniLM`则可在CPU上良好运行。
- **开启批处理**：使用`model.encode(texts)`等原生方法一次性处理多条文本，能充分利用GPU并行计算能力，显著提升吞吐量。
- **模型量化 (FP16/INT8)**：加载模型时设置`use_fp16=True`能**节省约一半的显存**且几乎不影响精度。
- **高效缓存 (推荐)**：针对重复内容，使用`functools.lru_cache`装饰器封装函数，可避免重复调用：将它加在一个函数上，可以自动缓存函数最近调用过的参数和对应的返回值，当相同的参数再次出现时，直接从缓存中读取结果，避免重复执行函数体。
  - 潜在注意事项
    **内存占用**
    每个缓存的 content 字符串和其对应的 768/1024 维向量（list[float]）都会保留在内存中。

    举例：一个 100 字符的字符串 + 1024 维向量 ≈ 8KB 内存。maxsize=10000 时，缓存大约占用 80MB 内存，可以接受。但如果文本很长（如 8192 tokens）且向量维度高，内存会显著增大。

    **字符串作为键**
    缓存区分完全相等的字符串。即使差一个空格或换行，也会视为不同参数，不会命中缓存。

    建议先对文本进行**规范化**（如去除多余空白、转小写、Unicode 归一化）再作为缓存输入。

    **不要在参数中传入不可哈希的对象**
    lru_cache 要求所有参数都是可哈希的（hashable）。str 是可哈希的，但 list、dict 不行。如果需要缓存列表参数，可以改为传入元组 tuple。

    **不要用于流式或非确定性函数**
    向量生成模型是确定性的（相同输入输出相同），非常适合缓存。但如果函数依赖随机种子或外部状态，就不能用 lru_cache。

    **缓存与异步**
    如果 generate_embedding_with_cache 是异步函数（async def），lru_cache 不能直接使用，需要改用 async_lru 库的 alru_cache。

  ```python
  from functools import lru_cache

  @lru_cache(maxsize=10000)
  def generate_embedding_with_cache(content: str) -> list[float]:
      # ... 你的向量生成逻辑 ...
      pass
  ```

- **异步执行**：在FastAPI等异步框架中，应将耗时的模型推理置于线程池中，避免阻塞主事件循环。

### ⚠️ 关键注意事项

1.  **模型下载**：首次使用时，所有本地模型都会自动从 Hugging Face Hub 下载到本地缓存（通常位于`~/.cache/huggingface/hub/`）。
2.  **维度一致性**：务必确保存入数据库的向量维度与查询时生成的维度一致，否则会出错。
3.  **硬件适配**：若使用GPU，请确保已安装对应版本的 PyTorch 和 CUDA。
4.  **显存与批次优化**：生成大量向量的内容时，注意监控显存使用，避免CUDA内存不足；可通过减小`batch_size`解决。
5.  **异步与线程池**：在FastAPI中，使用`asyncio.to_thread`或类似机制将同步的推理调用包装为异步，避免阻塞事件循环。
6.  **商业与合规检查**：使用第三方API或开源模型时，请遵守各服务商的使用政策与模型的开源许可协议。
